Reference Guide

Complete REST API documentation for the Food Delivery Platform — Phase 1. Covers authentication, restaurants, menu management, cart, orders, payments, reviews, and real-time delivery tracking.

Version v1.0 — Phase 1
Framework .NET 9 Web API
Database PostgreSQL / Supabase

Prepared by Jnaya LLC
01 — Overview

Introduction

This document describes every endpoint of the Food Delivery Platform REST API. It is intended for developers integrating mobile applications (Flutter — Android/iOS), customer web interfaces (Angular/Flutter Web), and admin dashboards. All responses are JSON. All timestamps are UTC ISO-8601.

Protocol
HTTPS + WSS
All traffic over TLS. Plain HTTP is redirected.
Content Type
application/json
Send Content-Type: application/json header.
Real-time
SignalR / WSS
Notifications and live delivery tracking via SignalR.
Pagination
page + pageSize
Query params for all list endpoints. Default: page=1, pageSize=10.
Image Storage
Cloudinary
Upload images via multipart/form-data endpoints.
Currency
INR (₹)
All monetary values are in Indian Rupees (decimal 18,2).

03 — Security

Authentication

All protected endpoints require a JWT Bearer token in the Authorization header. Tokens expire after 24 hours. Use the refresh endpoint to obtain a new access token without re-login.

ℹ
Authorization Header Format:
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
⚠
SignalR Connection: Pass the access token as a query parameter — ?access_token=YOUR_JWT — when connecting to SignalR hubs, since WebSocket headers are not supported by all clients.
04 — Authorization

Roles & Permissions

👑
SuperAdmin
Full platform access. Only role that can create Admin accounts and verify restaurants.
🏪
Admin
Restaurant owner. Manages their own restaurants, menus, and orders.
🛒
Customer
Browses restaurants, places orders, makes payments, and submits reviews.
🛵
DeliveryPartner
Receives deliveries, updates GPS location, and completes or fails orders.
05 — Errors

Error Handling

All errors follow a consistent ApiResponse<T> structure. HTTP status codes are meaningful — never 200 with an error body.

JSON — Error Response Schema 
{
  "success": false,
  "message": "Descriptive human-readable error message",
  "statusCode": 400,
  "data": null,
  "errors": ["Specific validation error 1", "Specific validation error 2"]
}
HTTP Code Meaning Common Cause
200 OK Successful GET, PATCH, PUT request.
201 Created Resource successfully created (POST).
400 Bad Request Validation failure, missing required fields, business rule violation.
401 Unauthorized Missing token, expired token, invalid credentials.
403 Forbidden Token valid but role insufficient for this action.
404 Not Found Resource ID does not exist or access is denied.
500 Server Error Unhandled exception. Logged via Serilog. Contact support.
06 — Testing

Swagger UI API Testing

The API ships with a fully configured Swagger UI available at the root URL in Development mode. It supports JWT authentication, detailed request/response schemas, and interactive endpoint testing.

https://localhost:7XXX/ (accessible at root in dev mode)

2

3
Authorize
Copy the accessToken from the response. Click the 🔒 Authorize button at the top. Enter Bearer YOUR_TOKEN.

POST /api/payment/initiate 🔒 Customer Create a Razorpay order for the placed order
Request
Response
Payment Flow
JSON — Request Body 
{ "orderId": "ord-uuid-here" }
JSON — Response 200 (pass to Razorpay SDK) 
{
  "success": true,
  "data": {
    "razorpayOrderId": "order_PxK9mQ2rT7vW",
    "amount": 562.90,
    "currency": "INR",
    "razorpayKeyId": "rzp_live_xxxxxxxxxxxx",
    "orderNumber": "FD-20250419-0001",
    "customerName": "Rahul Sharma",
    "customerEmail": "rahul.sharma@email.com",
    "customerPhone": "9876543210"
  }
}
ℹ
Razorpay Integration Flow:
1. Customer places order → POST /api/order
2. Call POST /api/payment/initiate → get razorpayOrderId & razorpayKeyId
3. Open Razorpay checkout in Flutter/Web SDK with the returned data
4. On success callback, call POST /api/payment/verify with the 3 Razorpay IDs
5. API verifies signature — order status changes to Confirmed
POST /api/payment/verify 🔒 Customer Verify payment signature after Razorpay checkout success
Request
Response
JSON — Request Body (from Razorpay callback) 
{
  "orderId": "ord-uuid-here",
  "razorpayOrderId": "order_PxK9mQ2rT7vW",
  "razorpayPaymentId": "pay_QaB3cD4eE5fF6g",
  "razorpaySignature": "9ef4b32a1c7d8f2e..."  // HMAC-SHA256 from Razorpay
}
JSON — Response 200 
{
  "success": true,
  "message": "Payment verified successfully."
  // Order status → Confirmed, PaymentStatus → Completed
  // Customer receives push notification
}

14 — Real-Time

SignalR Hubs

Connect using the SignalR client library. Pass your JWT as a query parameter. Both hubs require authentication.

ℹ
Flutter connection example:
HubConnectionBuilder().withUrl("https://api.yourdomain.com/hubs/notifications?access_token=YOUR_JWT").build()
🔔 Notification Hub
wss://api.yourdomain.com/hubs/notifications
ReceiveNotification Sent to user when any notification is created (order updates, payment, new order alerts for admin)
📍 Delivery Tracking Hub
wss://api.yourdomain.com/hubs/delivery
TrackOrder(orderId) Client method — call this to join the order group and start receiving live GPS updates
StopTrackingOrder(orderId) Client method — leave the order group when done tracking
LocationUpdate Server → Client. Fires when delivery partner pushes location. Contains orderId, latitude, longitude, timestamp.
